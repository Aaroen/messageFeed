# messageFeed 需求文档

## 1. 项目目标

`messageFeed` 是面向个人使用的信息聚合、受控 AI Agent、自然语言设置控制、主动网络采集、金融市场监控与 AI 摘要推送系统。系统从 RSS、Atom、JSON Feed、YouTube、微信公众号桥接源、网页变化源、主动网络搜索、金融行情源和后续 Agent 型来源采集内容，经过去重、排序、兴趣匹配、规则判断和大模型摘要后，通过 Feed 流、`messageFeed AI` 内部源、企业微信、`ntfy` 和其他通知通道向用户呈现。

当前部署边界限定为 WSL 本机单节点服务。外部访问已收敛为 Cloudflare Tunnel 域名入口 `https://aroen.eu.cc`，本机开发入口为 `https://localhost:8443`；局域网 IP 与 Tailscale IP 直连入口关闭。部署模式只表达拓扑形态，不表达访问范围；第一阶段默认拓扑为 `single_node`，访问范围由入口网关、端口绑定和 Cloudflare Tunnel 共同决定，公开访问基址由 `PUBLIC_BASE_URL` 决定。

设计阶段需要保留后续升级为分布式部署的接口，但分布式部署不作为第一阶段交付目标。后续目标是同一服务可以运行在本机、VPS 或其他配置机器上，并由 Cloudflare Tunnel 或 Cloudflare Load Balancer 将访问流量转发到健康节点。

第一阶段以 PostgreSQL 作为主存储，承载订阅源、条目、阅读状态、阅读行为事件、导入任务、通知记录、Agent 计划与审计、AI 生成条目元数据、主动网络采集快照、用户兴趣画像、金融行情快照和告警事件等需要事务、索引、唯一约束和可追溯查询的数据。Redis 不作为第一阶段主数据库；后续仅在缓存、任务队列、限流、短期状态或分布式锁存在明确收益时作为可选增强组件接入。

## 2. 目标用户与核心场景

- 用户希望集中订阅技术、财经、AI、新闻、博客、YouTube 频道和微信公众号等来源。
- 用户希望每天定时收到高价值内容摘要，而不是逐条阅读所有内容。
- 用户希望重大事件能够通过微信或其他推送通道及时到达。
- 用户希望可以从已有 RSS 阅读器导入 OPML，也可以从推荐源目录中快速选择订阅。
- 用户希望在 Web 界面中选择不同阅读方式：既可以按时间线从最新内容向较早内容浏览，也可以进入自动推荐 Feed，阅读来自已订阅源和未订阅候选源的条目。
- 用户希望在多端（Web、iOS、Android）使用统一的账号和数据，通过统一的 RESTful API 实现跨平台同步。
- 用户希望持续监控某个指数、股票、ETF 或其他金融标的，当当日涨跌幅超过设定阈值时，通过 AI Agent 解释可能背景，并通过微信等通道及时通知。
- 用户希望金融告警与资讯源联动，例如指数大幅波动时，同时参考当天财经新闻、政策消息和市场评论生成摘要。
- 用户希望通过自然语言让具备足够理解和规划能力的模型调整系统设置，例如增减订阅源、调整提示事件、订阅官方号文章、配置 AI 摘要提醒和修改通知偏好。
- 用户希望系统具备内置 AI 源，可以像阅读普通订阅源一样阅读日报、周报、热点事件分析、主动网络研究报告、金融分析和 Agent 执行结果。
- 用户希望 AI Agent 可以在没有 RSS 或稳定 API 的信息源中主动获取最新网络信息，例如官网公告页、产品更新页、政策公告页、财经事件页面和关键词搜索结果。
- 用户希望系统在尊重隐私和可解释性的前提下记录阅读行为、偏好标签和反馈，用于后续推荐、摘要、提醒和 Agent 决策。

## 3. MVP 功能范围

- 订阅源管理：新增、启用、停用、设置抓取周期、标签和权重。
- 源目录：提供内置推荐源，支持分类、关键词、语言、热度和站点筛选。
- 导入能力：支持 OPML 导入、URL 批量导入、从推荐源目录批量订阅。
- 内容抓取：拉取 RSS、Atom、JSON Feed，解析标题、链接、发布时间、摘要和正文片段。
- 内容去重：基于 `source_id + normalized_url`、原始 GUID 或内容哈希实现幂等入库。
- Feed 展示：按时间倒序返回内容，支持分页、来源过滤、已读、收藏和隐藏。
- Web 阅读模式：支持时间线模式和推荐 Feed 模式，用户可以在 Web 界面切换。
- 推荐 Feed：自动推荐用户可能需要阅读的条目，允许混合已订阅来源和未订阅候选来源。
- 兴趣规则：基于关键词、来源权重、标签和语言形成初步评分。
- AI 内部源：提供 `messageFeed AI` 虚拟来源，用于展示日报、周报、热点事件分析、主动网络研究报告、推荐内容包、金融分析和 Agent 操作报告。
- 主动网络采集：支持静态网页抓取、网页变化监控和搜索型采集，用于补足不支持 RSS、Atom、JSON Feed 或稳定 API 的来源。
- 阅读行为采集：记录条目曝光、打开、阅读进度、主动停留时间、点击原文、收藏、隐藏、不感兴趣和减少类似推荐等事件。
- 用户画像：基于显式偏好、隐式行为、短期兴趣、长期兴趣和负反馈形成可解释、可编辑、可回滚的偏好模型。
- AI 日报：按固定时间对高相关内容生成摘要、事件聚类和重点条目列表。
- 企业微信对话入口：MVP 优先支持企业微信自建应用接收消息 API，用户可以通过企业微信应用向系统提问并获得可审计回复。
- 重大事件推送：对高权重命中内容进行 AI 判断后触发通知，该能力在对话入口 MVP 之后接入。
- 通知通道：通知系统后置支持 `ntfy`、企业微信自建应用消息和可选智能机器人主动消息，不作为企业微信对话入口 MVP 的前置条件。
- 金融关注列表：支持新增、启用和停用指数、股票、ETF 等关注标的。
- 金融行情监控：支持定时拉取或订阅行情快照，至少记录当前价、前收价、当日涨跌幅、时间戳和数据源。
- 金融告警规则：支持当日涨跌幅绝对值超过阈值、价格上穿或下穿阈值、成交量异常等规则。
- AI Agent 金融告警：规则触发后调用 AI 生成简短解释，结合行情数据与相关资讯，明确标注“不构成投资建议”。
- 金融通知：金融告警通过微信、ntfy 等通道发送，并具备冷却时间、幂等键和触发历史。
- 项目级 AI Agent：用户可以用自然语言要求模型执行本项目内的受控智能操作，模型生成结构化意图和计划，经权限校验和必要确认后调用已注册能力执行。

## 4. 信息源需求

标准 RSS、Atom、JSON Feed 是第一优先级来源，使用 `gofeed` 解析。

推荐源目录借鉴 Folo 的发现源和 onboarding feed 机制。本地参考中 `references/rssnext_folo/apps/landing/public/discover-sources.json` 包含 1543 个发现源，`references/rssnext_folo/apps/mobile/src/modules/onboarding/feeds.json` 包含 236 个 onboarding 订阅源。实现时不能直接假定可无条件复制使用，需要记录来源出处、许可状态、健康状态和最近校验时间。

YouTube 优先使用频道 RSS 或 RSSHub 路由。若后续需要播放量、评论、搜索等结构化字段，再评估 YouTube Data API。

微信公众号内容源可通过 RSSHub、wechat2rss、AnyFeeder、Folo 候选源等方式补充，但必须标注桥接服务、稳定性和潜在失效风险。

X 不进入 MVP 强依赖。其 API 权限、费用、速率限制和访问稳定性均存在不确定性，应后续以独立 connector 评估。

对没有 RSS、Atom、JSON Feed 或稳定 API 的来源，系统应通过受控主动网络采集补足。主动采集优先级如下：

1. 静态网页抓取：直接 HTTP 获取 HTML，抽取标题、正文、发布时间和链接。
2. 网页变化监控：保存页面快照、正文 hash 和结构 hash，发现变化后生成条目或 AI 源报告。
3. 搜索型采集：按用户主题或 Agent 计划获取候选 URL，再抓取正文、去重和评估来源。
4. 渲染型采集：对强依赖 JavaScript 的页面使用受控 headless browser，后置实现。
5. 登录态或高限制采集：单独设计，不进入早期闭环。

主动采集必须保存来源 URL、抓取时间、抽取方式、内容 hash、失败原因和稳定性评价。搜索结果不能直接视为事实，必须经过抓取、去重和来源评估后才能进入摘要、推荐或通知。

## 5. Web 阅读视图与推荐 Feed 需求

### 5.1 前后端架构与多端支持

#### 5.1.1 架构选型决策

采用**前后端分离 + 统一 API** 架构：

**选型理由**：
1. **多端需求**：需支持 Web、iOS、Android，统一 API 便于扩展。
2. **独立开发**：前后端可独立迭代，提高开发效率。
3. **性能优势**：Go 后端性能优于 Node.js 全栈方案。
4. **类型安全**：OpenAPI 规范自动生成各端 SDK。
5. **部署简单**：Go 单一二进制，Vue 静态资源，便于静态资源分发。

**不采用 Next.js 全栈的原因**：
- Next.js API Routes 不适合作为独立 API 供移动端调用。
- 需要废弃已有 Go 代码并重写。
- Node.js 部署复杂度高于 Go 单一二进制。

#### 5.1.2 技术栈

**后端（Go）**：
- 语言：Go 1.26+
- HTTP 框架：Gin
- 数据库：PostgreSQL 15+
- 数据库迁移：`golang-migrate/migrate`，使用版本化 SQL `up/down` 文件
- API 规范：OpenAPI 3.0
- 文档：Swagger UI

**Web 前端（Vue 3）**：
- 框架：Vue 3.4+ (Composition API)
- 构建：Vite 8
- UI 库：Arco Design Vue 2.x（字节跳动开源，现代化设计）
- 状态管理：Pinia 2.x
- 路由：Vue Router 4
- HTTP：Axios（基于 OpenAPI 自动生成）
- 语言：TypeScript 5.x

**移动端（后续阶段）**：
- iOS：Swift + SwiftUI + Alamofire
- Android：Kotlin + Jetpack Compose + Retrofit

#### 5.1.3 API 设计规范

**统一前缀**：`/api/v1`

健康检查、就绪检查、Prometheus 指标和运行时节点信息不使用版本前缀，继续保留 `/healthz`、`/readyz`、`/metrics` 和 `/api/runtime/node`。业务 API 必须统一挂载在 `/api/v1` 下。

后端路由层使用 Gin。`*gin.Context` 只允许出现在 handler 和中间件层，service、repository、fetcher、domain 等模块统一使用 `context.Context`，避免业务逻辑绑定 HTTP 框架。

**通用响应格式**：
```json
{
  "code": 0,
  "message": "success",
  "data": {...},
  "request_id": "uuid"
}
```

**分页参数**：
- `limit`：每次返回数量（默认 20，最大 100）
- `offset`：从第几条记录开始返回（从 0 开始）
- 响应包含 `total` 总数

**错误处理**：
- HTTP 状态码：标准语义（200/400/401/404/500）
- 业务错误码：`code` 字段区分具体错误
- 错误详情：`message` 字段人类可读

##### 5.1.3.1 阶段二最小闭环要求

阶段二以“订阅源 -> 手动抓取 -> 去重入库 -> 时间线展示 -> 阅读状态”为最小闭环。该阶段必须先完成 Gin 路由迁移，并保持阶段一已有基础端点兼容。

阶段二后端 API 范围：

- `POST /api/v1/sources`：新增订阅源。
- `GET /api/v1/sources`：查询订阅源列表。
- `PATCH /api/v1/sources/{id}`：更新订阅源、启用或停用订阅源。
- `POST /api/v1/sources/{id}/fetch`：手动触发抓取。
- `GET /api/v1/items`：查询条目列表，支持分页、来源过滤、已读、收藏和隐藏过滤。
- `GET /api/v1/items/{id}`：查询条目详情。
- `POST /api/v1/items/{id}/read`：标记已读。
- `POST /api/v1/items/{id}/favorite`：标记收藏。
- `POST /api/v1/items/{id}/hide`：隐藏条目。
- `GET /api/v1/feed/timeline`：按时间线查询已订阅来源条目。
- `GET /api/v1/feed/view-mode`：读取 Web 阅读模式偏好。
- `PUT /api/v1/feed/view-mode`：保存 Web 阅读模式偏好。

阶段二数据范围包括 `sources`、`items`、`user_item_states` 和 `feed_view_preferences`。`sources` 使用启用、停用等状态表达用户管理动作；默认不要求物理删除订阅源。`items` 必须通过 `source_id + normalized_url`、非空 `raw_guid` 或内容哈希实现重复抓取幂等。

阶段二前端最小契约先写入 `api/openapi.yaml`，只覆盖前端立刻需要的条目列表、条目详情、订阅源列表和阅读模式偏好接口；其余接口随实现进展逐步补充。

阶段二本地联调使用统一 HTTPS 入口：浏览器访问 `https://localhost:8443` 或 `https://aroen.eu.cc`，Caddy 将 `/api`、`/healthz`、`/readyz`、`/metrics` 转发到 Go API，将页面请求转发到 Vite dev server。开发态不直接暴露 `5173` 和开发态 `60001`，生产部署继续沿用同域统一入口思路，避免浏览器跨域。

阶段二暂不交付源目录、OPML 导入、推荐 Feed、兴趣规则、周期调度、AI 摘要、通知和金融监控。这些能力分别归入后续阶段。

##### 5.1.3.2 最小业务闭环后的观测系统要求

阶段二完成“订阅源 -> 手动抓取 -> 去重入库 -> 时间线展示 -> 阅读状态”最小业务闭环后，下一阶段必须立即补齐完整的日志、错误追踪和请求链路观测系统，不得继续向源目录、推荐、AI 摘要、通知或金融监控扩展。

观测系统最低要求：

- 日志必须使用结构化 JSON 输出，保留 `service`、`environment`、`node_id`、`request_id`、`trace_id`、`span_id`、`method`、`path`、`status`、`duration_ms`、`error`、`operation` 等稳定字段。
- 应用日志默认输出到 stdout/stderr，由容器运行时或日志采集器负责落盘和轮转；本地开发可继续使用 Docker `json-file`，完整观测环境应接入 Loki 或等价日志查询系统。
- 观测系统必须有明确的磁盘保留策略：本地 Docker `json-file` 日志必须设置大小和文件数上限，Loki 必须配置保留期和过期删除，Prometheus 与 Tempo 必须配置或确认本地保留窗口。
- `/healthz`、`/metrics` 等高频成功探针请求不应在 info 级别持续写入业务访问日志；失败探针仍必须可追踪。
- 请求入口必须生成或继承 `X-Request-ID`，并将 request id 写入响应、日志和 `context.Context`，便于用户反馈错误时反查服务端记录。
- 链路追踪必须接入 OpenTelemetry。HTTP 入口、service、repository、fetcher、notifier、llm、scheduler 等关键边界应逐步建立 span，并通过 `trace_id` 与日志关联。
- 错误必须具备统一错误模型。业务错误、外部依赖错误、数据库错误和 panic 应由 handler 统一映射为 HTTP 状态码、业务错误码和用户可理解消息，同时在服务端保留错误链、操作名、request id 和 trace id。
- panic 必须由中间件恢复并记录 request id、trace id、method、path 和错误摘要；后续生产环境可接入 Sentry 或等价错误聚合系统，但不作为最小观测闭环的第一依赖。
- 指标必须覆盖 HTTP 请求量、耗时、错误率、数据库连接池、抓取次数、抓取失败、抓取耗时和外部调用耗时，并能通过 Grafana 或等价系统查看。

#### 5.1.4 前端 UI 设计

**设计原则**：
- 现代化、简洁、响应式
- 深色模式支持
- 移动端适配（响应式布局）

**主要页面**：
1. **订阅源管理** (`/sources`)：列表 + 新增/编辑表单 + 启用/停用开关；阶段二不提供删除按钮
2. **时间线模式** (`/timeline`)：瀑布流/卡片式展示
3. **推荐 Feed** (`/recommendations`)：智能推荐内容
4. **阅读详情** (`/items/:id`)：原文内嵌或摘要片段展示 + 阅读原文按钮 + 操作按钮
5. **设置页面** (`/settings`)：偏好配置

**核心组件**：
- `SourceList`：订阅源列表
- `FeedTimeline`：时间线视图
- `ItemCard`：内容卡片
- `ActionBar`：已读/收藏/隐藏操作

**主题色**：
- 主色调：蓝色系（#165DFF，Arco Design 默认）
- 深色模式：自动跟随系统或手动切换

Web 界面至少支持两种阅读模式：

1. 时间线模式：以用户已订阅来源为主，按发布时间倒序展示；当发布时间缺失时，以抓取时间作为排序兜底。用户可以从最新内容向较早内容连续浏览，并保留来源过滤、分页、已读、收藏和隐藏能力。
2. 推荐 Feed 模式：系统基于用户订阅、阅读行为、兴趣规则、来源质量和内容新鲜度自动推荐条目。推荐结果可以包含用户已订阅来源，也可以包含未订阅候选来源。

推荐 Feed 中的未订阅内容必须满足以下要求：

- 明确标注“未订阅来源”。
- 展示来源名称、来源出处、桥接方式和健康状态。
- 展示推荐原因，例如关键词匹配、相似主题、热门来源、同类来源或用户近期阅读偏好。
- 提供一键订阅、隐藏、减少类似推荐和反馈不感兴趣等操作。
- 不得因为用户阅读未订阅内容而自动订阅该来源。

推荐候选池来源：

- 用户已订阅来源的最新条目。
- 推荐源目录中的健康候选源。
- Folo onboarding/discover 候选源经过许可、健康和去重校验后的条目。
- RSSHub、YouTube RSS、微信公众号桥接等候选来源经过稳定性标注后的条目。

推荐排序应至少考虑：

- 内容发布时间或抓取时间。
- 用户兴趣规则、标签、关键词和来源权重。
- 用户阅读、收藏、隐藏和不感兴趣反馈。
- 来源健康状态、重复率和历史质量。
- 重大事件、财经波动或专题摘要上下文。

Web 界面应保存用户最近选择的阅读模式。推荐 Feed 的每条记录应可追溯推荐原因，避免形成不可解释的信息流。

阶段二条目详情页按不同信息源灵活处理，优先级如下：

1. 前端优先尝试 iframe 或等价内嵌方式展示原文 URL，类似 RSS 阅读器 WebView；该策略不要求后端改动，但会受原站 CORS、`X-Frame-Options` 和 CSP 限制。
2. 内嵌不可用时展示摘要片段和“阅读原文”按钮，作为阶段二默认兜底体验。
3. 后端保存全文和离线正文阅读作为后续增强，不进入阶段二默认范围，因为需要更大的存储层、清洗和安全策略改动。

内容安全要求：

- 后端条目响应提供 `content_text` 纯文本字段，供前端默认展示。
- `content_snippet` 可能包含外部来源 HTML，前端不得直接 `v-html` 渲染。
- 前端如确需展示 HTML，必须先经过 DOMPurify 或等价库净化，并继续保留“阅读原文”入口。

源管理交互要求：

- 阶段二不提供 `DELETE /api/v1/sources/{id}`。
- 前端用 Switch 表达启用、停用状态，后端通过 `PATCH /api/v1/sources/{id}` 设置 `status=active` 或 `status=inactive`。
- 后续如补充删除接口，应保留启用、停用作为默认安全操作；永久删除需要单独确认策略。

## 6. AI 源、阅读行为与用户画像需求

系统应提供一个内置虚拟来源 `messageFeed AI`。该来源不是外部 RSS，而是 AI Agent 生成内容的统一展示入口。用户可以像阅读普通订阅源一样阅读 AI 生成条目，并使用已读、收藏、隐藏、阅读原文、查看依据等通用阅读能力。

`messageFeed AI` 应至少支持以下内容类型：

- `daily_digest`：每日摘要。
- `weekly_digest`：每周摘要。
- `event_analysis`：热点事件分析。
- `web_research_report`：主动网络研究报告。
- `web_change_report`：非 RSS 网页变化报告。
- `recommendation_pack`：推荐内容包。
- `market_analysis`：金融市场分析。
- `source_health_report`：来源健康报告。
- `agent_action_report`：Agent 操作报告。

AI 源内容必须保留生成类型、输入内容范围、关联条目或网页快照、模型名称、提示词版本、token 消耗、生成时间、推送状态和用户反馈。金融分析类 AI 源内容必须包含“不构成投资建议”。

阅读行为采集应采用状态表与事件表分层：

- 状态层记录当前结果和累计统计，例如已读、收藏、隐藏、首次打开、最近打开、打开次数、最大阅读进度和累计主动停留时间。
- 事件层记录行为过程，例如曝光、打开详情、阅读进度、点击原文、收藏、隐藏、不感兴趣、减少类似推荐和通知打开。

停留时间只能统计页面可见、窗口聚焦且用户未切到后台的主动停留时间。单次停留时间应设置上限，避免误把离开页面后的挂起时间当成兴趣信号。系统不得采集鼠标轨迹、键盘输入轨迹、剪贴板、浏览器外部行为或无明确用途的高频原始滚动日志。

用户画像应服务于推荐、摘要、提醒和 Agent 决策，不作为黑箱广告画像。画像应区分：

- 显式偏好：用户主动设置或确认的主题、语言、来源类型、通知偏好。
- 隐式偏好：从阅读、收藏、隐藏、点击原文和停留时间中推断的偏好。
- 短期兴趣：最近数日或数周的临时关注。
- 长期兴趣：长期稳定偏好。
- 负反馈：不感兴趣、减少类似推荐、隐藏主题和低质量来源。

用户画像必须可解释、可编辑、可回滚。长期偏好不得由模型基于单次行为静默写入，应基于多次证据或用户确认。

用户画像同时是 Agent 的底层长期记忆之一。系统应允许 Agent 在每次执行前读取冻结的画像快照，作为推荐、摘要、提醒、搜索和订阅管理的背景信息；但原始阅读事件不应直接进入模型上下文，应先聚合为画像证据、近期兴趣或推荐解释。

画像写入要求：

- 显式偏好可以由用户主动设置、自然语言确认或设置页面修改产生。
- 隐式偏好应由多次行为证据聚合产生，证据包括阅读进度、主动停留时间、收藏、隐藏、不感兴趣、点击原文和通知打开。
- 长期兴趣必须保留证据链、置信度、更新时间和来源类型。
- Agent 执行失败、工具错误、采集失败或通知失败不得直接提升为用户偏好。
- 用户应能查看画像标签的形成依据，并能禁用、调低权重或回滚画像变更。

## 7. 金融市场监控需求

金融监控第一阶段以指数和少量用户关注标的为主，不进入自动交易、策略回测和组合管理。详细金融分析 Agent 方案见 `docs/financial-agent-plan.md`。

支持标的范围：

1. 指数：例如上证指数、沪深 300、创业板指、恒生指数、纳斯达克指数、标普 500。
2. 股票与 ETF：允许用户手动添加 symbol，并配置市场、币种、交易所和展示名称。
3. 后续扩展：基金、期货、外汇、加密资产等作为独立市场类型评估。

行情源应通过统一 `MarketDataProvider` 抽象接入。候选数据源包括 Yahoo Finance、Alpha Vantage、Finnhub、Polygon、Tushare、AkShare、东方财富、新浪财经等。具体实现时必须区分免费、延迟、实时、授权、速率限制和覆盖市场。

告警规则至少支持：

- `day_change_pct_abs_gte`：当日涨跌幅绝对值大于等于阈值。
- `day_change_pct_gte`：当日涨幅大于等于阈值。
- `day_change_pct_lte`：当日跌幅小于等于负阈值。
- `price_cross_above`：价格上穿指定值。
- `price_cross_below`：价格下穿指定值。
- `volume_ratio_gte`：成交量相对近期均值放大。

告警通知应包含标的、当前价、涨跌幅、触发阈值、行情时间、数据源、AI 简述和风险提示。AI 只在确定性规则触发后调用，不参与是否触发告警的基础判断。

成熟方案调研与扩展原则：

| 参考对象 | 主要启发 | 本项目采用方式 |
| --- | --- | --- |
| [QuantConnect LEAN](https://github.com/QuantConnect/Lean) | 证券模型、行情数据、市场日历、回测和实盘边界 | 只参考金融实体、数据源和市场日历边界，不引入完整交易引擎 |
| [TradingView Alerts](https://www.tradingview.com/support/solutions/43000520149-introduction-to-tradingview-alerts/) | 价格条件、指标条件、Webhook、触发频率控制 | 抽象为 `market_alert_rules`，先实现涨跌幅、价格穿越和成交量放大 |
| [Grafana Alerting](https://grafana.com/docs/grafana/latest/alerting/) 与 [Alertmanager](https://prometheus.io/docs/alerting/latest/alertmanager/) | 规则状态、通知路由、静默、分组、去重 | 告警事件必须具备状态、冷却时间、通知目标、`dedupe_key` 和历史记录 |
| [AkShare](https://akshare.akfamily.xyz/)、[Tushare](https://tushare.pro/)、[Alpha Vantage](https://www.alphavantage.co/documentation/)、[Finnhub](https://finnhub.io/docs/api)、[Massive/Polygon](https://massive.com/docs) | 多市场行情 API、速率限制、授权与实时性差异 | 统一封装为 provider，不让业务层直接依赖具体行情源 |
| [go-talib](https://github.com/markcheno/go-talib) | 均线、RSI、MACD 等技术指标计算 | MVP 不引入复杂指标库，仅保留后续指标接口边界 |

金融数据源接入必须显式记录覆盖市场、是否实时、典型延迟、授权状态、速率限制、失败重试策略和字段完整性。Yahoo Finance、东方财富、新浪财经等非官方或网页逆向来源只能作为个人实验或原型候选，不得在产品文案中表述为稳定实时数据源。

金融能力的 MVP 只做“关注标的 -> 拉取行情 -> 规则命中 -> AI 解读 -> 微信或 ntfy 通知”。自动交易、券商账户接入、高频行情、完整回测、组合收益归因和确定性买卖建议均不进入本项目当前边界。

## 8. 企业微信对话入口与通知需求

系统应优先通过企业微信自建应用接收消息 API 建立 AI 对话入口，使用户可以在企业微信中向 `messageFeed AI Agent` 所属应用提问并收到可审计回复。每日摘要、重大事件提醒和任务执行结果的主动通知能力后置实现，避免在尚未建立 session、turn、权限和审计前引入主动打扰链路。

企业微信与微信通道优先级如下：

1. 企业微信自建应用接收消息 API：作为阶段五 P0 对话入口，负责接收用户消息、创建 Agent turn、生成回复和记录审计。
2. 企业微信自建应用消息发送：作为 P0 异步回复出口，并为后续正式通知出口提供 `access_token` 管理、可见范围校验和发送结果审计。
3. 企业微信网页授权及 JS-SDK：用于账号绑定、设置页和高风险操作确认；不作为聊天对话入口。
4. 企业微信智能机器人短连接或长连接：若后续管理台开放相关入口，可作为群聊 `@机器人` 或机器人原生交互的可选通道，当前不作为 P0 前提。
5. `ntfy`：作为阶段七通知系统的轻量推送通道。
6. 微信公众号测试号或正式公众号：适合正式用户触达，但配置和审核成本更高。
7. 个人微信桥接：仅作为实验性方案，不作为 MVP 的唯一交付路径。

企业微信自建应用对话入口最低要求：

- 服务必须具备公网 HTTPS 回调地址，当前可使用 Cloudflare Tunnel 域名 `https://aroen.eu.cc`。
- 企业微信管理台需要先创建自建应用，获取 `CorpID`、`AgentID`、`Secret`，并在应用内开启“接收消息”或“设置 API 接收”。
- 自建应用接收消息需要配置 URL、Token、EncodingAESKey；建议 URL 使用 `https://aroen.eu.cc/api/v1/channels/wechat-work/app/callback`。
- 应用可见范围必须包含当前用户；如企业微信要求配置可信 IP，则当前服务调用企业微信 API 的出口 IP 必须加入可信 IP。
- GET URL 验证必须处理 `msg_signature`、`timestamp`、`nonce`、`echostr`，校验签名并解密 `echostr`，在 1 秒内返回明文。
- POST 消息回调必须校验签名、解密 XML，解析 `FromUserName`、`MsgType`、`Content`、`MsgId`、`AgentID` 等字段，并优先使用 `MsgId` 做事件排重。
- 回调处理应快速入库并返回，模型调用与回复发送由后台 worker 执行，避免企业微信 5 秒超时和三次重试导致重复处理。
- 异步回复使用自建应用 `message/send`，文本内容不超过 2048 字节，且 `touser`、`toparty`、`totag` 不能同时为空。
- 网页授权及 JS-SDK 可信域名只能解决网页内身份授权、设置页交互和确认页能力，不能替代企业微信聊天窗口中的接收消息回调。
- P0 只允许只读能力：最近资讯查询、指定来源最新条目查询、当前消息摘要或简短问答。
- P0 不允许自动订阅、删除或停用来源、修改提醒、创建金融告警、主动推送通知或输出确定性投资建议。

企业微信对话体验最低要求：

- 用户可以在企业微信中直接向自建应用发送文本问题，并收到自然语言回复。
- 系统应支持“最近有什么值得看”“某个来源最近有什么更新”“总结这段内容”等只读请求。
- 系统应在无法识别来源、用户未绑定或没有足够上下文时返回明确的澄清问题或失败原因。
- 当模型处理较慢时，系统应先让企业微信回调成功结束，再通过应用消息异步返回结果，避免用户收到重复回复。
- 每次回复都必须可以关联到入站消息、session、turn、transcript、audit、request id 和 trace id。

企业微信任务执行最低要求：

- 查询、摘要和解释类任务可以在 P0 直接执行。
- 新增订阅、停用来源、调整抓取周期、创建提醒规则、配置通知、创建金融告警等会改变系统状态的任务，P0 只能生成待确认计划。
- 待确认计划必须包含目标对象、拟执行动作、影响范围、风险等级、幂等键、可回滚性和拒绝选项。
- 后续 P1 可通过 Web 确认页、企业微信确认流程或网页授权及 JS-SDK 身份确认完成用户确认，但确认后的实际变更仍必须由后端 service 执行。
- 泄露密钥、绕过访问限制、未授权通知目标、默认永久删除和未注册能力必须被拒绝。

通知记录必须保存触发原因、通道、接收目标、发送状态、失败原因、模型、token 和耗时信息。企业微信自建应用发送消息时，还必须保存 `invaliduser`、`invalidparty`、`invalidtag`、`unlicenseduser`、企业微信返回 `msgid`、request id 和 trace id。金融告警通知还必须保存行情快照、规则阈值、AI 解读版本和幂等键。

## 9. AI Agent 与自然语言设置控制需求

系统应支持由高能力模型通过用户自然语言控制产品设置和执行项目内受控智能操作。该能力定位为 `messageFeed AI Agent`，不是通用无限工具执行平台。模型可以在用户授权范围内操作系统已有设置和已注册能力，但不得绕过权限、审计、幂等和业务校验。完整 Agent 重组方案见 `docs/agent-plan.md`。

Agent 实现应采用 Go 后端内的轻量运行时，不以 fork 或裁剪 `OpenAI Codex`、`Claude Code` 等代码执行型 CLI Agent 为基础。成熟项目只作为架构参考，主要吸收 session/turn、工具路由、延迟能力发现、上下文窗口压缩、召回预算和权限决策机制。

覆盖原则：凡是产品提供给用户调整的配置项，原则上都应进入自然语言设置控制范围。后续新增订阅、摘要、提醒、通知、金融告警、来源目录、官方号桥接等配置能力时，必须同步声明其自然语言控制能力、风险等级、确认策略和回滚方式。

可控制设置范围：

1. 订阅源：新增、停用、恢复、调整抓取周期、标签、权重、分类和导入策略。
2. 推荐源与官方来源：从推荐源目录订阅来源；配置微信公众号、官方号、官方文章桥接源；记录来源出处、桥接方式、稳定性和失效风险。
3. 提示事件：新增、停用、恢复或调整重大事件提醒、关键词提醒、来源提醒、专题提醒和金融告警规则。
4. AI 摘要提醒：配置日报、周报、专题摘要、官方文章摘要、财经摘要和提醒时间。
5. 通知偏好：调整通知通道、接收目标、免打扰时间、冷却时间和消息模板。
6. 金融监控：新增或调整关注标的、阈值规则、冷却时间和通知目标。
7. 后续新增设置：任何新增用户可配置项都应注册到自然语言控制能力清单，除非存在明确安全、合规或成本原因。

Agent 控制流程：

1. 用户提出自然语言指令。
2. 系统创建或恢复 Agent session，并为本次输入创建 Agent turn；同一 session 同时只能有一个 active turn。
3. 系统构建当前任务的上下文和冻结记忆快照，读取用户画像、相关条目、AI 源报告、历史计划、执行边界和能力清单。
4. 模型将指令解析为结构化意图、目标对象、候选能力和风险等级。
5. 当核心能力不足时，系统应通过受控能力检索发现延迟能力，不应把全部能力 schema 长期注入模型上下文。
6. 系统生成 Agent 计划，展示目标对象、执行步骤、变更前后差异、影响范围、潜在风险和回滚方式。
7. 系统执行权限校验、配置校验、来源健康校验和成本估算，并将计划决策为 `allow`、`prompt` 或 `forbidden`。
8. 对新增通知、删除或停用大量订阅、提高通知频率、增加模型成本、修改金融告警等操作，默认要求用户确认。
9. 确认或授权通过后，系统通过已注册能力调用既有 service 接口执行，不允许模型直接写数据库。
10. 系统保存原始指令、Agent turn、Agent 计划、执行结果、模型版本、提示词版本、用户确认记录、审计日志、上下文归档、记忆召回记录和必要的 AI 源报告。

Agent 上下文与记忆要求：

- 活动上下文不是长期存储。系统应保留原始 transcript、压缩摘要、结构化记忆和可检索归档，避免依赖单次模型窗口保存历史。
- 当上下文估算达到模型窗口的 50% 到 60% 时，应触发压缩或归档评估；压缩后目标区间为 30% 到 40%。
- 当前目标、待确认问题、最新用户指令、当前计划、近期关键工具结果、显式新增偏好和安全策略属于不可压缩保护区。
- 归档只能按完整语义块执行，不得把同一决策、工具调用和工具结果拆散到不同归档块中。
- 压缩摘要必须保留原始归档引用、时间范围、语义类型、关联计划、来源条目和可信等级。
- 需要历史信息时，Agent 应通过受控回忆工具检索归档、AI 源报告、画像证据、相关条目或历史计划。
- 召回内容属于不可信上下文，只能作为证据输入，不得覆盖系统规则、权限策略、用户确认策略或能力边界。
- 召回工具应区分 search、preview 和 full get。全文召回必须记录召回原因、使用位置和预算消耗；高风险操作不得只依赖摘要或 preview。

执行模式：

- `suggest_only`：模型只给出建议和变更计划，不实际执行。
- `approval_required`：模型可以生成计划，用户确认后执行；该模式作为默认模式。
- `delegated`：用户预先授权后，模型可以在明确策略内自动执行低风险设置变更。

能力暴露模式：

- `core`：少量基础能力默认可见。
- `deferred`：通过能力搜索按需发现和加载 schema。
- `hidden`：仅后端策略或已确认计划内部可调用。

Agent 评估要求：

- 系统应维护固定 Agent 评测集，覆盖订阅管理、推荐与画像、AI 源生成、主动网络采集、通知、金融分析、上下文记忆和安全对抗。
- 每个评测用例应记录初始用户画像、现有订阅、现有条目、可用能力、用户输入、预期计划、预期权限决策、预期工具调用、预期状态变更和禁止行为。
- Agent 评测执行必须保存 transcript、结构化计划、工具调用、状态差异、AI 源输出、审计日志、模型版本、提示词版本、token、成本和耗时。
- 自动评分应优先使用确定性规则和数据库状态断言；LLM-as-judge 或人工复核只用于文本质量、事实一致性和解释质量。
- 评测指标至少覆盖任务成功率、工具选择准确率、权限决策正确率、高风险操作确认率、越权操作拦截率、事实引用完整率、召回准确率、用户画像证据覆盖率、通知误报率、通知漏报率、成本和耗时。
- 涉及 prompt injection、敏感信息、未授权通知目标、默认永久删除和绕过访问限制的安全对抗用例必须作为回归门禁。

风险边界：

- 模型能力不足、意图歧义或对象匹配不唯一时，必须向用户追问，不得猜测执行。
- 物理删除订阅、通知通道、告警规则等持久数据不作为默认行为；自然语言中的“删除”优先解释为停用或归档，除非用户明确要求永久删除并二次确认。
- 密钥、token、Webhook URL 等敏感信息不得暴露给模型明文上下文。
- 模型不得自行创建绕过权限的通知目标，不得向未授权接收人发送消息。
- 所有设置变更必须可查询、可审计，并尽可能支持回滚。

## 10. 非功能需求

- 可部署：第一阶段即提供 Docker Compose，至少包含服务本体、PostgreSQL 和一次性数据库迁移服务。
- 远程访问：第一阶段通过 Tailscale 访问本机服务，服务监听地址、端口和公开访问基址必须可配置；`DEPLOYMENT_MODE` 不得被用于推导监听范围。
- 主存储：第一阶段以 PostgreSQL 作为唯一主存储；Redis 仅作为后续缓存、队列、限流、短期状态或任务锁实现的可选接口预留。
- 数据库迁移：PostgreSQL 初始化目录不得直接挂载完整 `migrations` 目录；迁移必须通过 `golang-migrate/migrate` 记录版本和 dirty 状态，API 启动时只检查迁移状态，不自行修改 schema。
- 可观测：第一阶段暴露 `/healthz`、`/readyz`、`/metrics`；最小业务闭环完成后必须立即补齐结构化日志存储、错误追踪、OpenTelemetry 链路追踪、日志与 trace 关联、核心 Dashboard 和告警入口。
- 可验收：提供 `make verify`，覆盖格式、测试、构建和关键集成检查。
- 安全：所有密钥从环境变量或外部配置注入，禁止硬编码。
- 可替换：业务层不直接依赖 RSSHub、ntfy、微信 SDK 或 OpenAI SDK。
- 单用户优先：第一阶段可单用户运行，但数据表预留 `user_id`。
- 分布式预留：API 服务应尽量无状态；配置中预留 `node_id`、`deployment_mode`、`public_base_url`；定时任务和通知发送需要保留任务锁与幂等接口，避免后续多节点部署时重复执行。
- 金融合规边界：系统只提供信息监控和摘要，不提供买卖建议、自动交易或收益承诺；金融通知必须保留风险提示。
- Agent 控制边界：自然语言设置变更和项目内智能操作必须通过结构化计划、权限校验、用户确认策略、能力注册和审计日志，不允许模型直接修改数据库或绕过业务服务。
- 主动采集边界：主动网络采集必须保存来源、时间、hash、抽取方法和失败原因；搜索结果不能直接视为事实；登录态采集、绕过访问限制和规避反爬不进入早期实现。
- 用户画像边界：用户画像必须可解释、可编辑、可回滚；不得采集无明确用途的高敏行为数据；长期偏好不能由单次行为静默确认。

## 11. 不在 MVP 的范围

- 通用无限工具执行 Agent 平台。
- 无边界浏览器自动化采集。
- 登录态采集、绕过访问限制和规避反爬。
- X 深度接入。
- Cloudflare 负载均衡和多节点故障转移。
- 自动交易、下单、券商账户接入。
- 完整量化回测平台。
- 高频行情和低延迟交易基础设施。
- 独立向量数据库。
- 多租户权限体系。
- 移动端原生应用。

## 12. 业务验收标准

- 可以新增一个 RSS 源并手动抓取内容。
- 可以从 OPML 导入多个源，并返回成功与失败明细。
- 可以从推荐源目录搜索并批量订阅来源。
- 可以查询 Feed 列表和条目详情。
- 可以在 Web 界面切换时间线模式和推荐 Feed 模式。
- 时间线模式可以按发布时间从最新到较早展示已订阅条目。
- 推荐 Feed 可以展示已订阅和未订阅来源的推荐条目，并标注推荐原因和订阅状态。
- 可以标记已读、收藏和隐藏。
- 可以记录条目打开、点击原文、阅读进度、主动停留时间和推荐反馈等阅读行为事件。
- 可以在 `messageFeed AI` 源中查看日报、热点事件分析、主动网络研究报告或 Agent 执行报告。
- 可以对一个无 RSS 的网页执行主动采集或变化监控，并将结果写入普通条目或 AI 源报告。
- 可以查看并编辑用户兴趣标签，且能看到关键偏好的形成依据。
- 可以生成一次日报摘要。
- 可以通过 `ntfy` 或微信发送测试通知。
- 可以通过 `make compose-up` 启动 PostgreSQL、完成 pending 数据库迁移、启动 API 并通过健康检查。
- 可以在 Tailscale 网络内通过配置的本机地址访问 API。
- 可以添加一个指数关注规则，例如“沪深 300 当日涨跌幅绝对值大于等于 2%”。
- 可以拉取该指数行情快照并计算当日涨跌幅。
- 当规则命中时，可以生成 AI 解读并通过微信或 ntfy 发送金融告警。
- 可以用自然语言生成设置变更计划，例如“帮我订阅 Go 官方博客和某公众号文章，每天早上摘要提醒”。
- 用户确认后，系统可以执行该计划并保存审计记录。
- 对高风险或歧义设置变更，系统会要求用户确认或补充信息。
