  总体目标
  将当前流程从“后端关键词规则决定工具范围”调整为：

  用户消息 -> 主 Agent 理解规划 -> 后端权限/预算校验 -> 简单任务直接回答或派发子 Agent -> 子 Agent 执行工具 -> 主 Agent 评估证据 -> 证据不足则迭代重规划 -> 最终回复

  一、必须修改的后端文件

  1. internal/agent/planner.go

  职责：生成 AgentPlan 和 AgentPlanStep。

  现状：selectCapabilities() 通过硬编码关键词判断 web.search、conversation.query_history、agent.schedule_task 等能力。

  拟改：

  - 保留后端 policy 裁决。
  - 去掉“关键词直接决定工具”的主路径。
  - 接收主 Agent 的结构化规划结果，再生成 plan/steps。
  - 将 planner metadata 记录为 main_agent_planning、complexity、required_capabilities、subagent_context 等。

  调用关系：
  AgentConversationService.createPlanForTurn() -> Planner.Build() -> repository.CreateAgentPlan()。

  2. 拟新增 internal/agent/plan_spec.go

  职责：定义主 Agent 结构化规划模型。

  建议类型：

  - PlanSpec
  - PlanSubtask
  - EvidenceRequirement
  - PlanningComplexity
  - PlanningDecision

  拟承载字段：

  - 用户意图
  - 任务类型
  - 复杂度
  - 是否需要子 Agent
  - 需要的能力
  - 子 Agent 提示词
  - 证据要求
  - 最大迭代次数
  - 最终回答约束

  调用关系：
  AgentConversationService 调用主 Agent 规划 -> 得到 PlanSpec -> 交给 Planner.BuildFromSpec()。

  3. internal/service/agent_conversation_service.go

  职责：整个 Agent 闭环编排中枢。

  现状：
  processTurn() 创建 controller run -> 创建 plan -> turnRunner.Run() 一次执行 -> 回填步骤 -> 发送最终回复。

  拟改：

  - 在 createPlanForTurn() 前后增加“主 Agent 理解与规划”阶段。
  - 记录主 Agent 规划 prompt、输出、解析结果。
  - 后端做权限、预算、能力 scope 校验。
  - 简单任务：允许主 Agent 直接回答。
  - 复杂任务：派发子 Agent 执行工具。
  - 执行后调用主 Agent 证据评估。
  - 证据不足时执行一次或有限次数重规划。
  - 最终企业微信回复只保留结果、依据、分析，不暴露执行治理细节。
  - Web 所需细节写入 plan metadata / run context trace。

  调用关系：
  handler -> ReceiveWeChatWorkAppMessage() -> processTurn() -> createControllerRun() -> createPlanForTurn() -> turnRunner.Run() -> bindPlanStepsToObservations() ->
  sendWeChatWorkFinalReportDelivery()。

  4. internal/agent/runner.go

  职责：模型对话、工具调用、最终回答生成。

  现状：模型只能调用 AllowedToolKeys 内的工具；scope 由 planner 预先决定。

  拟改：

  - 接收主 Agent 合成的子 Agent 上下文。
  - 区分 controller prompt 与 executor prompt。
  - 在工具执行后返回更结构化的 ContextSnapshot。
  - 为主 Agent 证据评估提供 observations、artifacts、tool results。
  - 继续保留 scope 校验，防止模型越权调用工具。

  调用关系：
  AgentConversationService.processTurn() -> TurnRunner.Run() -> generateReply() -> chatWithTools() -> executeToolCall()。

  5. internal/service/agent_run_recording_executor.go

  职责：子 Agent 工具执行过程记录。

  现状：每次工具调用创建 executor run，并记录 tool_input、executor_output、observation、artifact。

  拟改：

  - executor run 的 task_packet 写入子 Agent 提示词、上下文摘要、迭代编号。
  - 记录工具调用参数、观察、产物、错误、重试信息。
  - 为 Web 流水线“子 Agent 默认收起、展开查看细节”提供完整数据。

  调用关系：
  TurnRunner.executeToolCall() -> agentRunRecordingExecutor.CallTool() -> agentP0CapabilityExecutor.CallTool()。

  6. internal/service/agent_runtime_adapters.go

  职责：具体工具实现，例如 feed、web search、web fetch、repo search、schedule。

  拟改：

  - 搜索浏览能力增强。
  - 减少“业务关键词固化”对结果筛选和结论生成的影响。
  - 工具只负责获取、过滤基础质量、返回证据，不负责替主 Agent 做结论。
  - 搜索失败、空结果、解析失败要形成清晰 observation，而不是最终回复空洞化。

  调用关系：
  agentRunRecordingExecutor -> agentP0CapabilityExecutor.CallTool() -> webSearch() / webFetchPage() / queryRecentItems() 等。

  7. internal/agent/answer_quality.go

  职责：回答质量与证据充分性检查。

  现状：已有较多金融市场方向校验逻辑，容易变成领域特化规则。

  拟改：

  - 改为通用证据充分性框架。
  - 金融类只作为 domain-specific optional rule，不作为所有问题的主规则。
  - 输出“证据充分 / 不充分 / 需要重查”的结构化结果。

  调用关系：
  TurnRunner.generateReply() -> applyAnswerQualityGate() -> EvaluateAnswerQuality()。

  8. internal/agent/evidence_score.go

  职责：证据相关性、质量评分。

  拟改：

  - 从固定关键词匹配转向通用评分字段：相关性、来源质量、时效性、覆盖度、冲突程度。
  - 给主 Agent 评估阶段提供可解释数据。

  调用关系：
  answer_quality.go 和 runner.go 从 observations/artifacts 中提取证据后调用。

  9. internal/agent/task_spec.go

  职责：任务规格识别。

  拟改：

  - 主路径改为模型输出的 PlanSpec。
  - 保留少量安全默认值，避免模型规划失败时无能力可用。

  调用关系：
  当前由 runner.go、agent_runtime_adapters.go 调用；后续只作为 fallback。

  四、后端完整调用链

  1. 企微回调进入 wechat_work_handler.go。
  2. handler 调用 AgentConversationService.ReceiveWeChatWorkAppMessage()。
  3. service 创建/更新 inbound message、session、turn。
  4. 异步模式下立即回复“后台任务已开始”。
  5. processTurn() 创建 controller run。
  6. 主 Agent 理解用户任务，生成结构化 PlanSpec。
  7. 后端根据 registry、policy、budget 校验能力范围。
  8. Planner 根据 PlanSpec 创建 AgentPlan 和 AgentPlanStep。
  9. 简单任务直接回答；复杂任务调用 TurnRunner。
  10. TurnRunner 使用子 Agent prompt 和 allowed scopes 调工具。
  11. agentRunRecordingExecutor 创建 executor run，记录工具输入、输出、观察、产物、错误。
  12. bindPlanStepsToObservations() 将 observation 回填到 plan step。
  13. 主 Agent 评估证据充分性。
  14. 证据不足时按预算重规划并再执行。
  15. 生成最终回答。
  16. 企微只发送面向用户的结果；Web 通过 plan/run/detail API 查看完整过程。
